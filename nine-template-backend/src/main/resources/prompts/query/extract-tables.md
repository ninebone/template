### ⚠️ 핵심 추론 가이드라인 (Reasoning Rules)
1. **비즈니스 관계성 추론**: 사용자가 요구하는 데이터가 단독 테이블로 해결되지 않는 경우, 데이터 모델 관점에서 부모/자식 관계나 매핑(Mapping) 테이블, 이력(History) 테이블이 징검다리로 끼어들어야 하는지 족보를 철저히 추적하십시오.
    - *예시: 사용자가 '회원의 주문 내역과 배송 상태'를 요구했다면, `TB_USER`와 `TB_ORDER`뿐만 아니라 이 둘을 연결하는 매핑 테이블이나 배송 상세 테이블(`TB_DELIVERY`)까지 함께 패키징하여 누락 없이 추출해야 합니다.*
2. **환각(Hallucination) 방지**: [전체 테이블 목록]에 존재하지 않는 가상의 테이블명이나 추측성 오브젝트는 절대로 지어내거나 포함하지 마십시오.

### 🎯 출력 형식 (Output Format)

다음 마크다운 양식을 엄격히 준수하여 답변을 작성하세요.
자바 파이프라인 엔진이 정상적으로 파싱할 수 있도록 `[TARGET_TABLES]` 섹션에는 테이블명 외의 다른 문자나 설명을 섞지 마십시오.

---

#### [TARGET_TABLES]
누락 없는 필수 타겟 테이블 목록을 오직 쉼표(,)로만 구분하여 한 줄로 나열하세요.
(예시: TB_USER_INFO, TB_ORDER_MASTER, TB_ORDER_DETAIL)

#### [JOIN_STRATEGY_GUIDE]
선택한 테이블들을 최종 쿼리에서 어떻게 조인(JOIN)하고 결합해야 하는지 비즈니스 로직과 외래키(FK) 추론 관계를 3줄 이내로 간결하게 기술하십시오.
(예시: TB_USER_INFO의 USER_ID와 TB_ORDER_MASTER의 USER_ID를 기준으로 1:N 조인하고, 주문 상세 조회를 위해 TB_ORDER_DETAIL을 ORDER_NO 기반으로 추가 결합함.)
